00 · Старт

Суть всего гайда — на одном экране

Твоя задача — не «знать все CSS-свойства», а не утонуть в хаосе, когда проект растёт. Поэтому здесь основной вопрос такой: что именно должно стать глобальным инструментом, а что — остаться локальным стилем компонента.

01
Повторяется — выноси
Кнопки, инпуты, брейкпоинты, flex-хелперы, контейнеры — это не стили одной страницы. Это общий слой проекта.
02
Уникально — держи рядом
Лейаут карточки, анимация конкретного меню, сетка одной страницы — пиши в Component.module.scss рядом с компонентом.
03
SCSS — инструмент, не магия
SCSS нужен для модулей, миксинов и удобства. Но цвета, радиусы, отступы и темы лучше держать в CSS-переменных.
04
Адаптив начинай с мобилки
Базовые стили = мобильная версия. Дальше слой за слоем добавляй улучшения на больших экранах.
i
Главная мысль

Один файл = одна ответственность. Если в одном файле одновременно брейкпоинты, кнопки, типографика, формы и карточки — это уже начало болота.

01 · Фундамент

Архитектура файлов

У проекта два мира: глобальные инструменты и локальные стили компонентов. Ниже — простой и рабочий скелет для React + Vite.

src/
├── app/
│ └── styles/
│ ├── abstracts/
│ │ ├── _breakpoints.scss
│ │ ├── _variables.scss
│ │ ├── _mixins.scss
│ │ ├── _typography.scss
│ │ └── index.scss
│ ├── base/
│ │ ├── _reset.scss
│ │ ├── _global.scss
│ │ └── _fonts.scss
│ ├── components/
│ │ ├── _button.scss
│ │ ├── _input.scss
│ │ └── _container.scss
│ └── index.scss
│
└── components/
└── ProductCard/
├── ProductCard.jsx
├── ProductCard.module.scss
└── index.js

Три правила, которые держат систему чистой

RULE 01
Partial-файлы начинаются с _
Так Sass понимает, что это не отдельный entrypoint, а вспомогательный модуль.
RULE 02
abstracts сами CSS не генерят
Их задача — дать проекту инструменты. Готовые классы живут в components/ или в локальных module-файлах.
RULE 03
Глобальный импорт один раз
src/app/styles/index.scss импортируется один раз в main.jsx. Не тащи глобальные стили по компонентам.
02 · Модули

@use и @forward вместо @import

!
Старый синтаксис уходит

@import устарел. Новый стандарт — @use и @forward. С ними видно, откуда пришёл каждый миксин или токен, и один и тот же модуль не грузится бесконтрольно снова и снова.

×Плохо · всё в глобале
@import './variables';
@import './mixins';
@import './breakpoints';

.btn {
  background: $primary;
  @include respond-to(md) {
    padding: 16px 24px;
  }
}
Непонятно, где живёт $primary и кто дал respond-to(). Любой импорт может всё переопределить.
✓Хорошо · модульно
@use 'sass:map';
@use 'app/styles/abstracts' as *;

.btn {
  background: var(--color-primary);

  @include respond-to('md') {
    padding: 16px 24px;
  }
}
Источник зависимостей явный. Никакой «магии» и неожиданных конфликтов имён.

@forward — фасад папки

Смысл очень простой: в одной папке лежат несколько partial-файлов, а index.scss делает их единым публичным API.

abstracts/index.scss
@forward 'breakpoints';
@forward 'variables';
@forward 'mixins';
@forward 'typography';
components/_button.scss
@use '../abstracts' as *;

@mixin button-base {
  display: inline-flex;
  align-items: center;
  justify-content: center;
}

.btn { @include button-base; }
ProductCard.module.scss
@use 'app/styles/abstracts' as *;

.card {
  @include flex-col(12px);
  @include respond-to('md') {
    padding: 24px;
  }
}
★
Современная привычка

Привыкай также к встроенным Sass-модулям: @use 'sass:map', @use 'sass:math', @use 'sass:color'. Старые глобальные built-in функции тоже постепенно уходят.

03 · Миксины

Миксины: умно, а не везде подряд

Миксин — это шаблон CSS. Он удобен, но важно понимать нюанс: при каждом @include Sass копирует итоговые декларации в CSS. Поэтому миксин полезен, когда он правда описывает повторяемую идею, а не когда им заменяют вообще всё.

×Плохо · свалка
// _mixins.scss
@mixin flex-center { ... }
@mixin button-primary { ... }
@mixin input-base { ... }
@mixin title-xl { ... }
@mixin card-shadow { ... }
@mixin modal { ... }
@mixin hero-grid { ... }
// и ещё 70 штук
Найти нужный кусок через месяц почти нереально. Файл превращается в свалку и источник дублирования.
✓Хорошо · по зонам ответственности
abstracts/
  _breakpoints.scss
  _mixins.scss
  _typography.scss
  _variables.scss

components/
  _button.scss
  _input.scss
  _container.scss
Нужны кнопки — идёшь в _button.scss. Нужны брейкпоинты — в _breakpoints.scss. Логика прозрачна.

Базовый набор, которого реально хватает

abstracts/_breakpoints.scss
@use 'sass:map';

$breakpoints: (
  'sm': 640px,
  'md': 768px,
  'lg': 1024px,
  'xl': 1280px,
);

@mixin respond-to($point) {
  @if map.has-key($breakpoints, $point) {
    @media (min-width: map.get($breakpoints, $point)) {
      @content;
    }
  } @else {
    @warn "Unknown breakpoint: #{$point}";
  }
}
abstracts/_mixins.scss
@mixin flex-center {
  display: flex;
  align-items: center;
  justify-content: center;
}

@mixin flex-col($gap: 0) {
  display: flex;
  flex-direction: column;
  gap: $gap;
}

@mixin line-clamp($lines: 2) {
  display: -webkit-box;
  -webkit-line-clamp: $lines;
  -webkit-box-orient: vertical;
  overflow: hidden;
}

@mixin hover {
  @media (hover: hover) and (pointer: fine) {
    &:hover { @content; }
  }
}
components/_button.scss
@use '../abstracts' as *;

@mixin button-base {
  display: inline-flex;
  align-items: center;
  justify-content: center;
  gap: 8px;
  padding: 12px 20px;
  border: 0;
  border-radius: var(--radius-md);
  cursor: pointer;
  font: inherit;
  transition: background .2s, transform .1s, box-shadow .2s;

  &:focus-visible {
    outline: none;
    box-shadow: 0 0 0 3px var(--color-primary-soft);
  }
}

@mixin button-primary {
  @include button-base;
  background: var(--color-primary);
  color: var(--color-on-primary);

  @include hover {
    background: var(--color-primary-hover);
  }
}

.btn {
  @include button-base;

  &--primary { @include button-primary; }
  &--full { width: 100%; }
}
components/_input.scss
@use '../abstracts' as *;

@mixin input-base {
  width: 100%;
  padding: 12px 16px;
  border: 1px solid var(--color-border);
  border-radius: var(--radius-md);
  background: var(--color-surface);
  color: var(--color-text);
  font: inherit;
  transition: border-color .2s, box-shadow .2s;

  &:focus {
    outline: none;
    border-color: var(--color-primary);
    box-shadow: 0 0 0 3px var(--color-primary-soft);
  }
}

.input { @include input-base; }
.input--error { border-color: var(--color-danger); }

Ключевая тонкость: не дублируй тяжёлый CSS без причины

×Плохо · 20 локальных копий одной кнопки
// в 20 разных module.scss
.orderBtn { @include button-primary; }
.saveBtn  { @include button-primary; }
.payBtn   { @include button-primary; }
.buyBtn   { @include button-primary; }
Итоговый CSS разрастается, а по факту это один и тот же визуальный паттерн.
✓Хорошо · глобальный класс + локальные отклонения
<button className="btn btn--primary">Купить</button>

// только если нужен частный кейс:
.orderBtn {
  @include btn.button-primary;
  width: 100%;
}
Базовый системный стиль — глобально. Локальный миксин — только когда нужен действительно особый вариант.
04 · Токены

CSS-переменные vs SCSS-переменные

Это один из самых важных рубежей. SCSS-переменные существуют на этапе сборки. CSS-переменные живут в браузере и могут меняться в рантайме.

SCSS
$breakpoints, карты, вычисления
Используй для всего, что служит самому SCSS: карты брейкпоинтов, флаги, промежуточные вычисления, имена внутри модулей.
CSS
--color-primary, --radius-md
Используй для токенов интерфейса: цвета, отступы, радиусы, тени, z-index, размеры, которые может захотеться переключать или переопределять темой.
i
Смотри на этот файл

Кнопка «Тема» в этом документе переключает цветовую схему без перекомпиляции. Это и есть сила CSS-переменных вживую.

×Плохо · всё на SCSS
$primary: #E85FA3;
$bg: #0A0A0F;

.btn {
  background: $primary;
}

// светлая тема?
// придётся пересобирать CSS
Для тем, runtime-смены, кастомизации через JS такой подход слишком жёсткий.
✓Хорошо · токены в :root
:root {
  --color-primary: #E85FA3;
  --color-bg: #0A0A0F;
  --color-text: #F2EFE7;
}

[data-theme='light'] {
  --color-bg: #FFFFFF;
  --color-text: #0A0A0F;
}

.btn { background: var(--color-primary); }
Тема меняется мгновенно. И сам CSS становится гибче и чище.
app/styles/base/_global.scss
:root {
  --color-primary: #E85FA3;
  --color-primary-hover: #FF6DAE;

05 · Адаптив

Адаптив: mobile-first навсегда

Если ты никогда раньше не делал адаптив, запомни одну главную схему: сначала мобилка, потом расширение. Не огромный десктопный CSS с кучей откатов вниз, а маленькая база плюс надстройки на больших экранах.

mobile-first пример
.grid {
  display: grid;
  grid-template-columns: 1fr;
  gap: 12px;

  @include respond-to('sm') {
    grid-template-columns: repeat(2, 1fr);
  }

  @include respond-to('lg') {
    grid-template-columns: repeat(3, 1fr);
    gap: 24px;
  }

  @include respond-to('xl') {
    grid-template-columns: repeat(4, 1fr);
  }
}
×Плохо · desktop-first
.card {
  grid-template-columns: repeat(3, 1fr);

  @media (max-width: 1023px) {
    grid-template-columns: repeat(2, 1fr);
  }

  @media (max-width: 639px) {
    grid-template-columns: 1fr;
  }
}
Нижние брейкпоинты всё время «отменяют» верхние. Логика сложнее, конфликтов больше.
✓Хорошо · mobile-first
.card {
  grid-template-columns: 1fr;

  @include respond-to('md') {
    grid-template-columns: repeat(2, 1fr);
  }

  @include respond-to('lg') {
    grid-template-columns: repeat(3, 1fr);
  }
}
Мобильная база читается первой. Большие экраны получают только добавки.

Три приёма, которые снимают половину боли

01
clamp() для текста и отступов
Часто можно обойтись без нескольких медиа-запросов. Размер растёт плавно между минимальным и максимальным значением.
02
auto-fit / minmax() для карточек
Грид сам раскладывает карточки по колонкам без ручного перечисления каждого состояния окна.
03
Container queries, когда компонент живёт в разных колонках modern
Если компонент должен реагировать на ширину своего родителя, а не всего окна, контейнерные запросы — сильный вариант.
container query пример
.cardWrap {
  container-type: inline-size;
  container-name: product-card;
}

.card {
  display: flex;
  flex-direction: column;
}

@container product-card (min-width: 500px) {
  .card {
    flex-direction: row;
  }
}
06 · Скоупы

Глобальное vs локальное · где что писать

Это главный вопрос в твоей задаче. Да: локальные компоненты логично описывать в .module.scss. И да: общие кнопки, инпуты и контейнеры логично держать в глобальном стиле/миксинах — но делать это нужно без фанатизма.

Этот стиль нужен больше чем одному компоненту?
ДА
Глобально
Пиши миксин или глобальный класс в src/app/styles/.

Примеры: .btn, .input, .container, respond-to(), heading-xl.
НЕТ
Локально
Пиши Component.module.scss рядом с компонентом.

Примеры: необычная сетка секции, частная анимация, особый layout карточки, стиль конкретного hero-блока.

Как это выглядит в React

ProductCard.jsx
import styles from './ProductCard.module.scss';

export const ProductCard = ({ product }) => (
  <article className={styles.card}>
    <img className={styles.image} src={product.image} alt="" />
    <h3 className={styles.title}>{product.title}</h3>
    <button className="btn btn--primary btn--full">
      В корзину
    </button>
  </article>
);

★
Практическое правило

Системный UI — глобально. Локальная композиция — модульно. Гибрид — берёшь глобальный миксин как базу и донастраиваешь в локальном module-файле.

i
React-нюанс

В реальном приложении над глобальными стилями кнопки или инпута часто лежит ещё и React-компонент <Button /> или <Input />. Это даже лучше: логика и разметка становятся единообразными, а стиль всё равно остаётся системным.

07 · Практика

Как всё собрать в React + Vite

Ниже — минимальный, но уже взрослый поток использования: глобальный импорт в main.jsx, глобальные классы для UI и локальные module.scss для композиции.

src/main.jsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import App from './App.jsx';
import './app/styles/index.scss';

createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>
);

@use 'base/global';

@use 'components/button';
@use 'components/input';

// abstracts сюда не подключаются:

★
Что здесь важно

В JSX глобальные классы вроде btn пишутся строкой. Локальные классы — через объект styles. Это две разные системы, и они должны жить спокойно рядом.

08 · Конфиг

Настройка Vite без ловушек

Здесь есть важное современное уточнение: для Sass в Vite лучше думать не только про алиасы, но и про то, как сам Sass ищет файлы для @use. Самый спокойный путь — JS/TS алиасы оставить для JavaScript, а для SCSS дать Sass load path до src.

1 · Установка

терминал
npm install -D sass-embedded

# если нужно, fallback-вариант:
# npm install -D sass

2 · Конфиг

vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'node:path';

export default defineConfig({
  plugins: [react()],

  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
    },
  },

  css: {
    preprocessorOptions: {
      scss: {
        loadPaths: ['src'],

        // Для старых Vite 5.4/5.x, если видишь legacy warning:
        // api: 'modern-compiler',
      },
    },
    modules: {
      generateScopedName: '[name]__[local]__[hash:base64:4]',
      localsConvention: 'camelCaseOnly',
    },
  },
});

3 · Как теперь импортировать

✓JS / TS
import { ProductCard } from '@/components/ProductCard';
import App from '@/app/App.jsx';
Для JS/TS алиас @ прекрасен и действительно избавляет от ../../../.
✓SCSS через loadPaths
@use 'app/styles/abstracts' as *;
@use 'app/styles/components/button' as btn;
Sass спокойно ищет файлы от src. Для @use это надёжнее, чем слепо рассчитывать на alias-магии.
!
Не начинай с additionalData

Автоматическая подмешка @use во все файлы через additionalData кажется удобной, но она прячет зависимости. Для командной разработки и дебага лучше явные импорты. Если когда-нибудь и использовать additionalData, то очень осознанно и только для минимальных токенов.

09 · Антипаттерны

Частые ошибки начинающего фронтендера

01
Слишком глубокая вложенность
.card .header .title .icon быстро взрывает специфичность. Старайся не уходить глубже 2–3 уровней.
02
Магические числа и цвета хардкодом
Не раскидывай #E85FA3 и 18px по проекту. Для UI-токенов используй CSS-переменные.
03
Hover без проверки устройства
На тач-экранах :hover ведёт себя не так, как на мышке. Лучше оборачивать hover через специальный миксин.
04
Один огромный mixins.scss
Такой файл неизбежно превращается в свалку. Разбивай по ответственности: кнопки, формы, типографика, брейкпоинты.
05
Глобальные классы без системы
.title или .item глобально — плохая идея. Либо CSS Modules, либо нормальные системные имена вроде .btn--primary.
06
Миксин вместо каждой повторяющейся штуки
Не всё, что повторяется, обязано быть миксином. Иногда правильнее глобальный класс или React-компонент.
07
Desktop-first по привычке
Если сначала описал огромный desktop layout, потом будешь мучительно резать его для мобилки. Начинай снизу.
08
Слепое копирование старых гайдов
@import, глобальные функции, неявная магия через сборщик — это то, на чём легко застрять в прошлом.
10 · Шпаргалка

Шпора, которую можно держать рядом при вёрстке

Ниже — короткая опора. Если забыл, куда писать стиль, просто пробеги глазами по этому блоку.

шаблон любого module.scss
@use 'app/styles/abstracts' as *;

.root {
  @include flex-col(16px);
  padding: 16px;
  background: var(--color-surface);
  border-radius: var(--radius-lg);

  @include respond-to('md') {
    padding: 24px;
  }

  @include hover {
    transform: translateY(-2px);
  }
}
Если
используется один раз
Пиши в Component.module.scss рядом с компонентом.
Если
повторяется по проекту
Выноси в src/app/styles/abstracts или src/app/styles/components.
Если
это цвет/радиус/spacing
Скорее всего это CSS-переменная, а не SCSS-переменная.
Если
это брейкпоинт
Скорее всего это карта в _breakpoints.scss и миксин respond-to().
мини-шпора по импортам
// module.scss
@use 'app/styles/abstracts' as *;
@use 'app/styles/components/button' as btn;

// jsx
import styles from './Component.module.scss';

// глобально в main.jsx
import './app/styles/index.scss';

Глобальная системная кнопка
global class + mixin
Куда класть
src/app/styles/components/_button.scss
Как использовать
className="btn btn--primary"
Почему
Это системный UI-элемент, он должен быть единым по приложению.

Если базовая кнопка нужна по всему приложению, лучше сделать её глобальным классом и/или React-компонентом, а миксины использовать как внутреннюю строительную базу.

Рекомендуемый шаблон
@use '../abstracts' as *;

@mixin button-primary {
  @include button-base;
  background: var(--color-primary);
}

.btn--primary {
  @include button-primary;
}
