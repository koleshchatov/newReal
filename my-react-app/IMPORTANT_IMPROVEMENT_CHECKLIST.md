## [ ] 3. После create/edit/delete обновлять данные на экране, а не ждать перезагрузку

### Что за проблема простыми словами

Сейчас список ролей и пользователей загружается, но после создания, редактирования и удаления store нормально не обновляется.
Из-за этого интерфейс легко отстает от настоящего состояния сервера.

### Почему это реально проблема, а не придирка

Это один из самых неприятных видов багов: "на сервере одно, на экране другое".
Пользователь перестает понимать, чему верить.

### Где это видно в проекте

- `src/Pages/RolePage/roleSlice.jsx`
- `src/Pages/PersonalPage/userSlice.jsx`

Сейчас в `extraReducers` обрабатывается загрузка списка, но почти не обрабатываются успешные `create/edit/delete`.

### Когда это станет большой проблемой

Когда один и тот же объект начнут менять несколько людей.
Один пользователь увидит старое состояние, другой новое.
В админке это быстро приводит к повторным действиям, путанице и неправильным решениям.

### Практический совет, как чинить

Для новичка самый надежный путь такой:

- после успешного `create/edit/delete` заново запрашивай список;
- только когда проект станет стабильнее, можно оптимизировать и обновлять store точечно.

Это проще и надежнее, чем сразу пытаться вручную синхронизировать все кейсы.

### Пример, как исправить

```jsx
async function deletePost() {
  try {
    await dispatch(deleteRolePost({ token, code: modal.data.code })).unwrap();
    await dispatch(roleList(token)).unwrap();
    dispatch(closeModalRole());
  } catch (error) {
    console.error(error);
  }
}
```

Либо можно обрабатывать это прямо в `extraReducers`, но для старта повторный запрос списка обычно безопаснее.

### Самопроверка

- [x] После создания новой роли она сразу видна в таблице без перезагрузки страницы.
- [x] После удаления запись сразу исчезает.
- [x] После редактирования видны новые данные, а не старые.
- [x] Если операция упала, список не притворяется обновленным.

## [ ] 4. Собрать хранение токена и session-логики в одном месте

### Что за проблема простыми словами

Сейчас работа с токеном и `deviceId` разбросана по разным местам:

- где-то чтение из `localStorage`;
- где-то запись;
- где-то удаление;
- где-то логика обновления токена.

Это значит, что важная логика авторизации живет кусками.

### Почему это реально проблема, а не придирка

Когда auth-логика размазана по проекту, ее очень легко случайно сломать.
Один файл очищает токен по-своему, другой записывает по-своему, третий ожидает другой формат.

Последствия:

- странные вылеты из сессии;
- баги "после логина пустой экран";
- забыли очистить токен или deviceId в одном месте;
- тяжело менять схему авторизации.

### Где это видно в проекте

- `src/auth/authSlice.jsx`
- `src/Pages/AuthPage/index.jsx`

### Когда это станет большой проблемой

Когда появятся:

- автообновление токена;
- несколько защищенных страниц;
- обработка `401`;
- logout по таймауту;
- разные окружения и правила безопасности.

### Практический совет, как чинить

Сделай один маленький модуль, например `src/auth/session.js`, и все операции с токеном и `deviceId` держи только там.

Правило:

- компонент не работает с `localStorage` напрямую;
- с хранилищем работает только session-модуль.

### Пример, как исправить

```js
const TOKEN_KEY = "token";
const DEVICE_ID_KEY = "deviceId";

export const session = {
  getToken() {
    return localStorage.getItem(TOKEN_KEY);
  },

  setToken(token) {
    localStorage.setItem(TOKEN_KEY, token);
  },

  clearToken() {
    localStorage.removeItem(TOKEN_KEY);
  },

  getDeviceId() {
    let deviceId = localStorage.getItem(DEVICE_ID_KEY);

    if (!deviceId) {
      deviceId = crypto.randomUUID();
      localStorage.setItem(DEVICE_ID_KEY, deviceId);
    }

    return deviceId;
  },
};
```

Тогда в форме логина будет не `localStorage.deviceId`, а нормальный вызов:

```jsx
deviceId: session.getDeviceId(),
```

### Самопроверка

- [x] В компонентах больше нет прямых обращений к `localStorage`.
- [x] Вся работа с токеном идет через один session-модуль.
- [x] Логин, refresh и logout используют один и тот же источник правды.
- [x] После logout токен реально очищается везде одинаково.

## [x] 6. Убрать хрупкую верстку: fixed width, float, inline-style и магические отступы

### Что за проблема простыми словами

Сейчас в проекте много верстки, которая держится на "магии":

- `width: 400px`
- `height: 400px`
- `margin-left: 46px`
- `float: right`
- `100vh`
- inline-style прямо в JSX

Это работает, пока контент короткий, экран удобный, а текст не меняется.

### Почему это реально проблема, а не придирка

Такой интерфейс очень легко ломается:

- на мобильном;
- при масштабировании браузера;
- при длинных текстах;
- при новых кнопках;
- при переводе интерфейса;
- при появлении ошибок в форме.

### Где это видно в проекте

- `src/Pages/AuthPage/authPage.module.css`
- `src/сomponents/form/form.module.css`
- `src/сomponents/layout/index.jsx`
- `src/сomponents/menu/index.jsx`
- `src/Pages/PersonalPage/index.jsx`

### Когда это станет большой проблемой

Когда дизайнер или менеджер попросит:

- добавить еще одно поле;
- показать текст ошибки;
- сделать нормально на ноутбуке и мобильном;
- увеличить шрифт;
- открыть страницу на другом масштабе.

Тогда "ровная" верстка начнет разъезжаться.

### Практический совет, как чинить

Базовые правила:

- вместо `float` использовать `flex` и `grid`;
- вместо фиксированных размеров чаще использовать `min()`, `max()`, `clamp()`;
- вместо `100vh` для полноэкранных блоков использовать `100dvh`;
- inline-style переносить в классы;
- не выравнивать элементы через случайные `margin-left`.

### Пример, как исправить

Было:

```css
.main {
  width: 400px;
  height: 400px;
  margin: 7em auto;
}
```

Лучше:

```css
.main {
  inline-size: min(100%, 28rem);
  min-block-size: min(32rem, calc(100dvh - 2rem));
  margin-inline: auto;
  margin-block: clamp(1rem, 6vh, 4rem);
  padding: clamp(1rem, 2vw, 1.5rem);
}
```

Для панели действий лучше так:

```css
.pageHeader {
  display: flex;
  align-items: center;
  justify-content: space-between;
  gap: 1rem;
  flex-wrap: wrap;
}
```

### Самопроверка

- [x] На ширине `320px` интерфейс не разваливается.
- [x] При масштабе браузера `200%` кнопки и поля не наезжают друг на друга.
- [x] В JSX почти не осталось inline-style.
- [x] Для выравнивания используются `flex`/`grid`, а не `float` и случайные `margin-left`.

## [ ] 7. Починить доступность и HTML-семантику

### Что за проблема простыми словами

Сейчас некоторые элементы визуально выглядят нормально, но для браузера и вспомогательных технологий описаны плохо.

Примеры:

- `label` не связаны с `input`;
- кликабельные действия местами сделаны через `li`, а не через кнопку.

### Почему это реально проблема, а не придирка

Это влияет не только на людей со скринридером.
Последствия шире:

- хуже навигация с клавиатуры;
- автотесты сложнее писать;
- браузер хуже понимает форму;
- труднее поддерживать фокус и состояния;
- можно случайно сделать интерфейс неудобным даже для обычного пользователя.

### Где это видно в проекте

- `src/сomponents/button/button.jsx`
- `src/сomponents/icon/icon.jsx`
- `src/сomponents/menu/header.jsx`
- `src/Pages/AuthPage/index.jsx`
- `src/Pages/PersonalPage/index.jsx`

### Когда это станет большой проблемой

Когда:

- появится клавиатурная навигация;
- вы начнете писать e2e-тесты;
- кнопок и форм станет больше;
- в проекте понадобится стабильное поведение фокуса и форм.

### Практический совет, как чинить

Правила простые:

- у каждой кнопки есть понятное имя;
- у декоративной иконки `alt=""` и `aria-hidden="true"`;
- `label` связан с полем через `htmlFor` и `id`;
- кликабельное действие должно быть `button`, а не просто `li` или `div`.

### Пример, как исправить кнопку выхода

```jsx
export default function Button({
  children,
  className,
  onClick,
  disabled,
  ariaLabel,
}) {
  return (
    <button
      type="button"
      className={className}
      onClick={onClick}
      disabled={disabled}
      aria-label={ariaLabel}
    >
      {children}
    </button>
  );
}
```

```jsx
<Button

  onClick={handleClick}
  ariaLabel="Выйти из админ-панели"
>
  <img className={styles.iconExit} src={logoutIcon} alt="" aria-hidden="true" />
</Button>
```

### Пример, как исправить поле формы

```jsx
<label htmlFor="email">Email</label>
<input id="email" type="email" {...register("email")} />
```

## [ ] 8. Починить таблицу и меню действий по строкам

### Что за проблема простыми словами

Сейчас одна переменная `open` управляет меню действий для таблицы пользователей.
То есть состояние не привязано к конкретной строке.

Еще в таблице используется `rowIndex` как `key`.

### Почему это реально проблема, а не придирка

На маленьком проекте это может не бросаться в глаза.
Но как только список начнет меняться, сортироваться или фильтроваться, React может путать строки.

Последствия:

- открыл действия не у той записи;
- после обновления списка меню "переехало";
- React повторно использует не тот DOM-элемент;
- появляются странные UI-баги, которые трудно повторить.

### Где это видно в проекте

- `src/Pages/PersonalPage/index.jsx`
- `src/сomponents/form/form.jsx`

### Когда это станет большой проблемой

Когда в таблице станет больше записей и появятся:

- сортировка;
- фильтрация;
- пагинация;
- обновление списка после CRUD.

### Практический совет, как чинить

Нужно:

- хранить `openRowId`, а не просто `open`;
- использовать стабильный `key`, например `row.id`;
- меню действий вынести в отдельный компонент строки или отдельную кнопку-меню.

### Пример, как исправить

```jsx
const [openRowId, setOpenRowId] = useState(null);

function toggleRowMenu(rowId) {
  setOpenRowId((current) => (current === rowId ? null : rowId));
}
```

```jsx
<button className={styles.buttonEdit} onClick={() => toggleRowMenu(row.id)}>
  Действия
</button>;

{
  openRowId === row.id && <div className={styles.rowMenu}>...</div>;
}
```

И в таблице:

```jsx
<tr key={row.id}>
```

### Самопроверка

- [ ] Открытие меню одной строки не влияет на другие строки.
- [ ] После обновления списка действия остаются у правильной записи.
- [ ] В таблице не используется `rowIndex` как основной `key`.
- [ ] После сортировки или фильтрации поведение строки остается корректным.
