## [ х ] 1. Починить базовые правила React и сделать линт зеленым

### Что за проблема простыми словами

Сейчас часть компонентов названа с маленькой буквы: `authPage`, `personalPage`, `opportunitiesPage`.
Для React и eslint это выглядит не как компонент, а почти как обычная функция.
Из-за этого правила хуков считаются нарушенными.

### Почему это реально проблема, а не придирка

Это не вопрос вкуса.
Это базовое правило React.
Если такие вещи игнорировать, проект быстро превращается в код, которому нельзя доверять:

- линтер постоянно красный, и реальные ошибки теряются среди шума;
- новые разработчики не понимают, где настоящая проблема, а где "так исторически сложилось";
- при рефакторинге можно сломать хуки и словить трудноуловимые баги.

### Где это видно в проекте

- `src/Pages/AuthPage/index.jsx`
- `src/Pages/PersonalPage/index.jsx`
- `src/Pages/OpportunitiesPage/index.jsx`
- `src/Pages/NomenclaturePage/index.jsx`
- `src/Pages/auidePage/index.jsx`

### Когда это станет большой проблемой

Пока проект маленький, кажется, что "и так работает".
Но как только появятся новые хуки, условия, переиспользование логики и несколько разработчиков, вы перестанете понимать, где настоящая ошибка.
В итоге начнутся правки "наугад", а не инженерная работа.

### Практический совет, как чинить

Сделай одно простое правило для всего проекта:

- имя React-компонента всегда с большой буквы;
- неиспользуемые переменные и импорты удаляются сразу;
- после любого блока работы `npm run lint` должен быть зеленым.

### Пример, как исправить

Было:

```jsx
export default function authPage() {
  const dispatch = useDispatch();
  ...
}
```

Должно быть:

```jsx
export default function AuthPage() {
  const dispatch = useDispatch();
  ...
}
```

Еще один важный принцип:

```jsx
const { isLoadingAuth, error, authentication } = useSelector(
  (state) => state.auth,
);
```

Если `error` и `authentication` не используются, их надо убрать, а не оставлять "на потом".

### Самопроверка

- [x] `npm run lint` проходит без ошибок.
- [x] Во всех файлах React-компоненты названы с большой буквы.
- [x] В проекте не осталось неиспользуемых импортов и переменных.
- [x] После переименований ничего не сломалось в импортах.

## [x] 2. Нормально обработать ошибки API и не закрывать формы раньше времени

### Что за проблема простыми словами

Сейчас код часто ведет себя так, будто запрос прошел нормально, даже если сервер вернул ошибку.
Плюс модалки закрываются сразу после `dispatch`, еще до реального подтверждения успеха.

Для пользователя это выглядит так:

1. Нажал "Создать".
2. Окно закрылось.
3. Кажется, что все сохранилось.
4. Потом выясняется, что сервер вернул ошибку, а данные не сохранились.

### Почему это реально проблема, а не придирка

Это ломает доверие к интерфейсу.
Админка, которая говорит "все ок", когда ничего не сохранилось, опаснее, чем админка с явной ошибкой.

Последствия:

- пользователь повторно жмет кнопку и создает дубли;
- думает, что данные обновились, а на сервере старое значение;
- можно потерять изменения и не понять, что именно пошло не так.

### Где это видно в проекте

- `src/servises/utils.js`
- `src/Pages/RolePage/modalRoleContent/createRole.jsx`
- `src/Pages/PersonalPage/modalUserContent/createUser.jsx`
- `src/Pages/RolePage/modalRoleContent/editRole.jsx`
- `src/Pages/RolePage/modalRoleContent/deleteRole.jsx`

### Когда это станет большой проблемой

Как только появятся реальные пользователи, медленный интернет, ошибки авторизации, валидация на сервере, таймауты и нестабильный backend.
На локалке "вроде работает", а в реальности пользователь начнет терять данные и делать повторные операции.

### Практический совет, как чинить

Нужно сделать два правила:

- если `response.ok === false`, сервис обязан бросить ошибку через `throw`;
- модалка закрывается только после успешного `dispatch(...).unwrap()`.

### Пример, как исправить сервис

```js
export const fetchData = async ({
  path,
  urlParamsObject = {},
  options = {},
}) => {
  const queryString = qs.stringify(urlParamsObject);
  const query = queryString ? `?${queryString}` : "";
  const requestUrl = `${API_URL}${path}${query}`;

  const response = await fetch(requestUrl, {
    credentials: "include",
    ...options,
  });

  const responseData = await response.json().catch(() => null);

  if (!response.ok) {
    throw new Error(responseData?.message || "Ошибка запроса к серверу");
  }

  return responseData;
};
```

### Пример, как исправить форму

```jsx
async function createRoles(data) {
  try {
    await dispatch(
      createRole({
        token,
        code: data.code,
        name: data.name,
        description: data.description,
        isActive: data.isActive === "true",
      }),
    ).unwrap();

    dispatch(closeModalRole());
  } catch (error) {
    console.error(error);
  }
}
```

### Самопроверка

- [x] Если сервер вернул `400/401/500`, UI показывает ошибку и не делает вид, что все успешно.
- [x] Модалка закрывается только после реального успеха.
- [x] Ошибки не теряются внутри `fetchData`.
- [x] В сервисах не осталось лишних `console.log`.

## [ ] 3. После create/edit/delete обновлять данные на экране, а не ждать перезагрузку

### Что за проблема простыми словами

Сейчас список ролей и пользователей загружается, но после создания, редактирования и удаления store нормально не обновляется.
Из-за этого интерфейс легко отстает от настоящего состояния сервера.

### Почему это реально проблема, а не придирка

Это один из самых неприятных видов багов: "на сервере одно, на экране другое".
Пользователь перестает понимать, чему верить.

### Где это видно в проекте

- `src/Pages/RolePage/roleSlice.jsx`
- `src/Pages/PersonalPage/userSlice.jsx`

Сейчас в `extraReducers` обрабатывается загрузка списка, но почти не обрабатываются успешные `create/edit/delete`.

### Когда это станет большой проблемой

Когда один и тот же объект начнут менять несколько людей.
Один пользователь увидит старое состояние, другой новое.
В админке это быстро приводит к повторным действиям, путанице и неправильным решениям.

### Практический совет, как чинить

Для новичка самый надежный путь такой:

- после успешного `create/edit/delete` заново запрашивай список;
- только когда проект станет стабильнее, можно оптимизировать и обновлять store точечно.

Это проще и надежнее, чем сразу пытаться вручную синхронизировать все кейсы.

### Пример, как исправить

```jsx
async function deletePost() {
  try {
    await dispatch(deleteRolePost({ token, code: modal.data.code })).unwrap();
    await dispatch(roleList(token)).unwrap();
    dispatch(closeModalRole());
  } catch (error) {
    console.error(error);
  }
}
```

Либо можно обрабатывать это прямо в `extraReducers`, но для старта повторный запрос списка обычно безопаснее.

### Самопроверка

- [x] После создания новой роли она сразу видна в таблице без перезагрузки страницы.
- [x] После удаления запись сразу исчезает.
- [x] После редактирования видны новые данные, а не старые.
- [x] Если операция упала, список не притворяется обновленным.

## [ ] 4. Собрать хранение токена и session-логики в одном месте

### Что за проблема простыми словами

Сейчас работа с токеном и `deviceId` разбросана по разным местам:

- где-то чтение из `localStorage`;
- где-то запись;
- где-то удаление;
- где-то логика обновления токена.

Это значит, что важная логика авторизации живет кусками.

### Почему это реально проблема, а не придирка

Когда auth-логика размазана по проекту, ее очень легко случайно сломать.
Один файл очищает токен по-своему, другой записывает по-своему, третий ожидает другой формат.

Последствия:

- странные вылеты из сессии;
- баги "после логина пустой экран";
- забыли очистить токен или deviceId в одном месте;
- тяжело менять схему авторизации.

### Где это видно в проекте

- `src/auth/authSlice.jsx`
- `src/Pages/AuthPage/index.jsx`

### Когда это станет большой проблемой

Когда появятся:

- автообновление токена;
- несколько защищенных страниц;
- обработка `401`;
- logout по таймауту;
- разные окружения и правила безопасности.

### Практический совет, как чинить

Сделай один маленький модуль, например `src/auth/session.js`, и все операции с токеном и `deviceId` держи только там.

Правило:

- компонент не работает с `localStorage` напрямую;
- с хранилищем работает только session-модуль.

### Пример, как исправить

```js
const TOKEN_KEY = "token";
const DEVICE_ID_KEY = "deviceId";

export const session = {
  getToken() {
    return localStorage.getItem(TOKEN_KEY);
  },

  setToken(token) {
    localStorage.setItem(TOKEN_KEY, token);
  },

  clearToken() {
    localStorage.removeItem(TOKEN_KEY);
  },

  getDeviceId() {
    let deviceId = localStorage.getItem(DEVICE_ID_KEY);

    if (!deviceId) {
      deviceId = crypto.randomUUID();
      localStorage.setItem(DEVICE_ID_KEY, deviceId);
    }

    return deviceId;
  },
};
```

Тогда в форме логина будет не `localStorage.deviceId`, а нормальный вызов:

```jsx
deviceId: session.getDeviceId(),
```

### Самопроверка

- [x] В компонентах больше нет прямых обращений к `localStorage`.
- [x] Вся работа с токеном идет через один session-модуль.
- [x] Логин, refresh и logout используют один и тот же источник правды.
- [x] После logout токен реально очищается везде одинаково.

## [x] 5. Перестроить CSS: глобальные стили отдельно, CSS Modules отдельно

### Что за проблема простыми словами

Сейчас в CSS Modules лежат глобальные селекторы:

- `body`
- `input`
- `label`
- `a`
- `summary`
- `aside`
- `h3`

На вид кажется, что это "стили конкретного компонента", но на деле они утекают в весь проект.

### Почему это реально проблема, а не придирка

Это один из самых коварных CSS-багов.
Ты меняешь стиль логин-страницы, а потом внезапно ломаются поля в модалке пользователей.

То есть разработчик думает:

"Я поменял только `authPage.module.css`"

А по факту поменял внешний вид `label` или `a` вообще во всем приложении.

### Где это видно в проекте

- `src/Pages/AuthPage/authPage.module.css`
- `src/сomponents/form/form.module.css`
- `src/сomponents/layout/layout.module.css`
- `src/сomponents/menu/menu.module.css`

### Когда это станет большой проблемой

Когда в проекте появится больше страниц и больше людей.
Тогда каждое изменение CSS начнет ломать соседние экраны.
Такие баги особенно раздражают, потому что ломается "не там, где меняли".

### Практический совет, как чинить

Раздели стили на два слоя:

- `globals.css` или `theme.css` для `body`, `a`, `input`, переменных, reset;
- `*.module.css` только для классов конкретного компонента.

### Пример, как исправить

В `src/main.jsx`:

```jsx
import "./styles/globals.css";
```

В `src/styles/globals.css`:

```css
:root {
  --color-brand: #0958d9;
  --color-bg: #f5f7fb;
  --radius-md: 12px;
}

body {
  margin: 0;
  background: var(--color-bg);
  font-family: "Segoe UI", Tahoma, Geneva, Verdana, sans-serif;
}

input,
button,
textarea,
select {
  font: inherit;
}
```

А в `authPage.module.css` оставляй только классы вроде `.main`, `.submit`, `.field`.

### Самопроверка

- [x] В `*.module.css` не осталось глобальных селекторов вроде `body`, `input`, `label`, `a`.
- [x] Глобальные переменные и базовые стили вынесены в один отдельный файл.
- [x] После правки стилей одной страницы не ломаются другие страницы.
- [x] Цвета, радиусы и отступы не копируются вручную по разным файлам.

## [x] 6. Убрать хрупкую верстку: fixed width, float, inline-style и магические отступы

### Что за проблема простыми словами

Сейчас в проекте много верстки, которая держится на "магии":

- `width: 400px`
- `height: 400px`
- `margin-left: 46px`
- `float: right`
- `100vh`
- inline-style прямо в JSX

Это работает, пока контент короткий, экран удобный, а текст не меняется.

### Почему это реально проблема, а не придирка

Такой интерфейс очень легко ломается:

- на мобильном;
- при масштабировании браузера;
- при длинных текстах;
- при новых кнопках;
- при переводе интерфейса;
- при появлении ошибок в форме.

### Где это видно в проекте

- `src/Pages/AuthPage/authPage.module.css`
- `src/сomponents/form/form.module.css`
- `src/сomponents/layout/index.jsx`
- `src/сomponents/menu/index.jsx`
- `src/Pages/PersonalPage/index.jsx`

### Когда это станет большой проблемой

Когда дизайнер или менеджер попросит:

- добавить еще одно поле;
- показать текст ошибки;
- сделать нормально на ноутбуке и мобильном;
- увеличить шрифт;
- открыть страницу на другом масштабе.

Тогда "ровная" верстка начнет разъезжаться.

### Практический совет, как чинить

Базовые правила:

- вместо `float` использовать `flex` и `grid`;
- вместо фиксированных размеров чаще использовать `min()`, `max()`, `clamp()`;
- вместо `100vh` для полноэкранных блоков использовать `100dvh`;
- inline-style переносить в классы;
- не выравнивать элементы через случайные `margin-left`.

### Пример, как исправить

Было:

```css
.main {
  width: 400px;
  height: 400px;
  margin: 7em auto;
}
```

Лучше:

```css
.main {
  inline-size: min(100%, 28rem);
  min-block-size: min(32rem, calc(100dvh - 2rem));
  margin-inline: auto;
  margin-block: clamp(1rem, 6vh, 4rem);
  padding: clamp(1rem, 2vw, 1.5rem);
}
```

Для панели действий лучше так:

```css
.pageHeader {
  display: flex;
  align-items: center;
  justify-content: space-between;
  gap: 1rem;
  flex-wrap: wrap;
}
```

### Самопроверка

- [x] На ширине `320px` интерфейс не разваливается.
- [x] При масштабе браузера `200%` кнопки и поля не наезжают друг на друга.
- [x] В JSX почти не осталось inline-style.
- [x] Для выравнивания используются `flex`/`grid`, а не `float` и случайные `margin-left`.

## [ ] 7. Починить доступность и HTML-семантику

### Что за проблема простыми словами

Сейчас некоторые элементы визуально выглядят нормально, но для браузера и вспомогательных технологий описаны плохо.

Примеры:

- кнопка выхода показывает только иконку;
- `title` передается в кнопку, но не используется;
- картинка в `Icon` без `alt`;
- `img` используется как будто у нее могут быть дочерние элементы;
- `label` не связаны с `input`;
- кликабельные действия местами сделаны через `li`, а не через кнопку.

### Почему это реально проблема, а не придирка

Это влияет не только на людей со скринридером.
Последствия шире:

- хуже навигация с клавиатуры;
- автотесты сложнее писать;
- браузер хуже понимает форму;
- труднее поддерживать фокус и состояния;
- можно случайно сделать интерфейс неудобным даже для обычного пользователя.

### Где это видно в проекте

- `src/сomponents/button/button.jsx`
- `src/сomponents/icon/icon.jsx`
- `src/сomponents/menu/header.jsx`
- `src/Pages/AuthPage/index.jsx`
- `src/Pages/PersonalPage/index.jsx`

### Когда это станет большой проблемой

Когда:

- появится клавиатурная навигация;
- вы начнете писать e2e-тесты;
- кнопок и форм станет больше;
- в проекте понадобится стабильное поведение фокуса и форм.

### Практический совет, как чинить

Правила простые:

- у каждой кнопки есть понятное имя;
- у декоративной иконки `alt=""` и `aria-hidden="true"`;
- `label` связан с полем через `htmlFor` и `id`;
- кликабельное действие должно быть `button`, а не просто `li` или `div`.

### Пример, как исправить кнопку выхода

```jsx
export default function Button({
  children,
  className,
  onClick,
  disabled,
  ariaLabel,
}) {
  return (
    <button
      type="button"
      className={className}
      onClick={onClick}
      disabled={disabled}
      aria-label={ariaLabel}
    >
      {children}
    </button>
  );
}
```

```jsx
<Button
  className={styles.buttonHeader}
  onClick={handleClick}
  ariaLabel="Выйти из админ-панели"
>
  <img className={styles.iconExit} src={logoutIcon} alt="" aria-hidden="true" />
</Button>
```

### Пример, как исправить поле формы

```jsx
<label htmlFor="email">Email</label>
<input id="email" type="email" {...register("email")} />
```

### Самопроверка

- [ ] Все кнопки можно понять без просмотра картинки.
- [ ] Все поля формы имеют связанный `label`.
- [ ] В `Icon` нет детей внутри `img`.
- [ ] Клавишей `Tab` можно пройти по основным действиям интерфейса в логичном порядке.

## [ ] 8. Починить таблицу и меню действий по строкам

### Что за проблема простыми словами

Сейчас одна переменная `open` управляет меню действий для таблицы пользователей.
То есть состояние не привязано к конкретной строке.

Еще в таблице используется `rowIndex` как `key`.

### Почему это реально проблема, а не придирка

На маленьком проекте это может не бросаться в глаза.
Но как только список начнет меняться, сортироваться или фильтроваться, React может путать строки.

Последствия:

- открыл действия не у той записи;
- после обновления списка меню "переехало";
- React повторно использует не тот DOM-элемент;
- появляются странные UI-баги, которые трудно повторить.

### Где это видно в проекте

- `src/Pages/PersonalPage/index.jsx`
- `src/сomponents/form/form.jsx`

### Когда это станет большой проблемой

Когда в таблице станет больше записей и появятся:

- сортировка;
- фильтрация;
- пагинация;
- обновление списка после CRUD.

### Практический совет, как чинить

Нужно:

- хранить `openRowId`, а не просто `open`;
- использовать стабильный `key`, например `row.id`;
- меню действий вынести в отдельный компонент строки или отдельную кнопку-меню.

### Пример, как исправить

```jsx
const [openRowId, setOpenRowId] = useState(null);

function toggleRowMenu(rowId) {
  setOpenRowId((current) => (current === rowId ? null : rowId));
}
```

```jsx
<button className={styles.buttonEdit} onClick={() => toggleRowMenu(row.id)}>
  Действия
</button>;

{
  openRowId === row.id && <div className={styles.rowMenu}>...</div>;
}
```

И в таблице:

```jsx
<tr key={row.id}>
```

### Самопроверка

- [ ] Открытие меню одной строки не влияет на другие строки.
- [ ] После обновления списка действия остаются у правильной записи.
- [ ] В таблице не используется `rowIndex` как основной `key`.
- [ ] После сортировки или фильтрации поведение строки остается корректным.

## [ ] 9. Почистить структуру проекта и убрать опасные мелочи в именах

### Что за проблема простыми словами

В проекте есть имена, которые потом обязательно выстрелят:

- `servises` вместо `services`;
- `auidePage` вместо `guidePage`;
- папка `сomponents`, где первая буква выглядит как латинская `c`, но это кириллица;
- лишние или дублирующие зависимости.

### Почему это реально проблема, а не придирка

Это тот тип проблем, который долго выглядит безобидно, а потом внезапно:

- ломает импорт;
- путает IDE;
- мешает поиску по проекту;
- делает onboarding медленнее;
- приводит к "у меня работает, у тебя нет".

### Где это видно в проекте

- `src/servises`
- `src/Pages/auidePage`
- `src/сomponents`
- `package.json`

### Когда это станет большой проблемой

Когда проект попадет:

- в CI;
- к другому разработчику;
- на другую машину;
- в массовый рефакторинг.

Особенно опасна папка с кириллической буквой: визуально все похоже, а фактически это другой символ.

### Практический совет, как чинить

Делай это отдельной аккуратной задачей:

- переименовать папки и файлы в понятные английские имена;
- обновить импорты;
- удалить неиспользуемые зависимости;
- после каждого шага гонять `lint` и `build`.

### Пример, как исправить

Что стоит привести к норме:

- `servises` -> `services`
- `auidePage` -> `GuidePage`
- `сomponents` -> `components`

По зависимостям отдельно проверить, действительно ли нужны все пакеты из `package.json`.

### Самопроверка

- [ ] Во всех путях используются ожидаемые латинские имена.
- [ ] После переименования все импорты обновлены.
- [ ] `npm run lint` и `npm run build` проходят.
- [ ] В `package.json` не осталось явно лишних библиотек.
